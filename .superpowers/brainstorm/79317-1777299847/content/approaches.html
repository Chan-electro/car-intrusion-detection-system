3 Architecture Approaches

Pick the one that fits your 3-day build window

A

Centralized MQTT Hub ★ Recommended

Single Mosquitto broker on laptop. ESP32 publishes all sensor data. Python ML server subscribes, detects intrusions, publishes corrections. Web dashboard shows live state. Hacker = second ESP32 or Python script injecting bad MQTT messages.

Pros

  • Buildable in 3 days
  • Easy to demo hacker injection
  • ML fully in Python (sklearn/keras)
  • Dashboard with Flask/FastAPI

Cons

  • No message authentication by default
  • Broker = single point of failure
B

Signed MQTT + Cryptographic IDS

Every MQTT message signed with HMAC-SHA256. Car rejects unsigned/bad-signature commands. GPS coordinates include timestamp + signature. Hacker cannot forge valid messages without the shared secret key.

Pros

  • Real cryptographic security
  • Hacker truly cannot override
  • Impressive for presentation

Cons

  • 3 days is tight
  • Key management overhead
  • Harder to demo intrusion visually
C

Hybrid: Edge Rules + Server ML

ESP32 runs local rule-based checks (speed delta threshold, IR signal hardwired). Server runs ML for complex patterns. Two-layer defense — local layer catches obvious attacks, server catches subtle ones.

Pros

  • Most realistic architecture
  • Car works even if server drops
  • Best for presentation narrative

Cons

  • More code across two layers
  • Sync/conflict logic needed
  • Riskiest for 3-day timeline
My recommendation: A — gets all 3 security scenarios working, ML running, dashboard showing, and hacker demo visible in 3 days. Add HMAC signing from B as a bonus if time allows on day 3.
