Scenario 1 — Traffic Signal (IR Hardware Lock)

HARDWARE DEFENSE
Attack Vector

Hacker publishes hacker/ir = GREEN to MQTT trying to keep car moving through red signal.

hacker publishes:
Topic: hacker/ir
Payload: {"signal": "green", "override": true}
Defense

Car firmware uses hardware interrupt on IR receiver pin. Software MQTT messages have zero authority over motor stop. Physical IR signal = physical stop.

// ESP32 firmware
attachInterrupt(IR_PIN, stopMotors, FALLING);
// MQTT hacker/ir topic is IGNORED
// Only physical IR triggers this
Flow
IR LED (38kHz pulse) → TSOP1738 detects
→ GPIO interrupt fires → PWM = 0 (stop)
→ MQTT publishes car/ir = RED
→ Dashboard shows STOPPED
→ IDS logs: "Traffic stop — legitimate"
Why Hacker Can't Win

MQTT is software. IR interrupt is hardware. No software path can override attachInterrupt() from network. Hacker would need physical access to the IR receiver wire to defeat this.

Scenario 2 — Speed Intrusion (ML Anomaly Detection)

ML DEFENSE
Attack Vector

Hacker publishes hacker/speed with abnormal value (e.g. 255 PWM = full throttle) trying to make car accelerate dangerously.

Defense

IDS server runs Isolation Forest on speed time-series. Spike outside learned normal range → anomaly → publish correction to car/command.

// ML Pipeline
Training data: 10 min normal driving → [pwm_val, delta_pwm, time_since_change]
Model: sklearn IsolationForest(contamination=0.05)
Input window: last 10 speed readings (100ms each = 1 second window)
// Detection loop
if model.predict([window]) == -1: # anomaly
  publish("ids/alert", {"type":"speed","severity":"HIGH"})
  publish("car/command", {"action":"set_speed","value": last_safe_speed})
Normal

PWM 60-140, smooth transitions, gradual accel/decel

Attack Pattern

PWM jumps to 255 in 1 reading. Delta = +115 in 100ms. Far outside training distribution.

IDS Response

Score < threshold → alert → restore last_safe_speed → log event

Scenario 3 — GPS Spoofing (Route Integrity Check)

RULE-BASED DEFENSE
Attack Vector

Hacker publishes hacker/gps with completely different lat/lon (e.g. teleport car to another city) to confuse routing.

Defense

Golden Route pre-loaded on server. Every 5s, IDS checks reported GPS vs expected next waypoint. Deviation > 50m → alert + route lock.

// GPS Integrity Service
GOLDEN_ROUTE = [(12.9716, 77.5946), (12.9720, 77.5950), ...] # waypoints
trusted_source = "car/gps" # only this topic is trusted
untrusted = "hacker/gps" # monitored but never used for routing

every 5s:
  car_pos = last_msg["car/gps"]
  expected = GOLDEN_ROUTE[current_waypoint_index]
  dist = haversine(car_pos, expected)
  if dist > 50: # meters
    publish("ids/alert", {"type":"gps_spoof","dist":dist})
    ROUTE_LOCKED = True # reject all external GPS updates
Key insight: IDS never reads from hacker/gps for routing. It only monitors it to detect attempted injection. The car's reported position (car/gps) is cross-checked against the pre-planned route — external topics cannot alter the golden route.
